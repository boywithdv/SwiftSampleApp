# デザイン

## このアプリで統一されたデザインを作成する

標準的なフォント・サイズ・色・スタックのスペーシング・パディング・角丸・アニメーションのタイミングなどを共有の定数 enum に配置し、すべての View から使用できるようにすることを優先する。これによりアプリのデザインが統一感のある一貫したものになり、簡単に調整できる。


## 柔軟でアクセシブルなデザインの要件

- 利用可能なスペースを読み取るために `UIScreen.main.bounds` を使用しない。`containerRelativeFrame()`・`visualEffect()` などの代替手段、または（代替がない場合に限り）`GeometryReader` を優先する。
- コンテンツが内側にきれいに収まる場合を除き、View に固定フレームを使用することを避ける。異なるデバイスサイズ・異なる Dynamic Type 設定などで問題が発生する可能性がある。フレームに柔軟性を持たせることが通常は好ましい。
- Apple の iOS でのインタラクションの最小許容タップ領域は 44x44 である。これを厳密に守ること。


## 標準的なシステムスタイリング

- データが不足または空の場合は、カスタムで何かを設計するのではなく、`ContentUnavailableView` を強く優先する。
- `searchable()` を使用する場合、`ContentUnavailableView.search` を使って空の結果を表示でき、使用した検索語が自動的に含まれる — `ContentUnavailableView.search(text: searchText)` のようなものを使う必要はない。
- アイコンとテキストを水平に並べる必要がある場合、`HStack` より `Label` を優先する。
- システムが適切なコンテキストに自動的に適応できるよう、手動のオパシティよりシステムの階層スタイル（例：secondary/tertiary）を優先する。
- `Form` を使用する場合、タイトルとコントロールが正しくレイアウトされるよう `Slider` などのコントロールを `LabeledContent` でラップする。
- `RoundedRectangle` を使用する場合、デフォルトの角丸スタイルは `.continuous` である — 明示的に指定する必要はない。


## すべての人に機能するデザインを確保する

- `fontWeight(.bold)` ではなく `bold()` を使用する。`bold()` を使用するとシステムが現在のコンテキストに適した正しいウェイトを選択できるためである。
- 重要な理由がある場合のみ bold 以外のウェイトに `fontWeight()` を使用する — `fontWeight(.medium)` や `fontWeight(.semibold)` を散りばめることは逆効果である。
- 明示的なリクエストがない限り、パディングとスタックのスペーシングにハードコードされた値を避ける。
- SwiftUI コードで UIKit の色（`UIColor`）を避ける。SwiftUI の `Color` またはアセットカタログの色を使用する。
- フォントサイズ `.caption2` は非常に小さく、一般的には避けるのが最善である。フォントサイズ `.caption` も小さめで、慎重に使用すべきである。
